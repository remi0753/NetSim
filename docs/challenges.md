# NetSim 課題集 — ネットワーク初学者のための構築チャレンジ

このシミュレーター（NetSim）で実際に手を動かしながら、L1〜L4 のネットワークを
段階的に学ぶための課題集です。難易度ごとに整理してあり、上から順にこなすと
「ケーブル1本」から「ミニデータセンター」まで自然にステップアップできます。

各課題は以下の形式で書かれています。

- **ねらい** — この課題で身につく概念
- **使う機能** — 必要なデバイス・コマンド
- **手順のヒント** — 詰まったときの道しるべ
- **✅ 達成条件** — 「できた」と判断するための確認方法

> 💡 全課題共通のコツ
> - デバイスを **ダブルクリック** するとコンソール（CLI）が開きます
> - 下部の **パケットログ** は行をクリックするとレイヤ別にデコード表示されます
> - 速度スライダ（x0.25〜x64）で、OSPF収束などは早送りすると快適です
> - ルータの新品インターフェースは `shutdown` 状態です。`no shutdown` を忘れずに

## 難易度の目安

| レベル | テーマ | レイヤ | 目安時間 |
|---|---|---|---|
| ⭐ Lv.0 | 操作に慣れる / 物理層 | L1 | 5〜10分 |
| ⭐⭐ Lv.1 | スイッチとMAC / VLAN | L2 | 各10分 |
| ⭐⭐⭐ Lv.2 | ルーティングの基本 | L3 | 各15分 |
| ⭐⭐⭐⭐ Lv.3 | 動的ルーティングと冗長化 | L3 | 各20分 |
| ⭐⭐⭐⭐⭐ Lv.4 | トランスポート層とサービス | L4 | 各20分 |
| 🏆 Lv.5 | 大規模・総合設計 | L1〜L4 | 30分〜 |
| 🧭 Lv.6 | 実務に近い境界・オーバーレイ設計 | L2〜L4 | 各30〜60分 |

> ⚠️ シミュレーター固有の前提
>
> - L2/L3スイッチは既定で **Rapid PVST+ 相当のSTP** が有効です。冗長なL2リンクを張ると、ループを起こす代わりに片方のポートが `blocking` になります。STPを無効化してブロードキャストストームを再現する機能はありません。
> - STPはトポロジ変更後ただちに収束します。実機のBPDUや listening / learning タイマーの待ち時間は再現しません。
> - OSPFは複数areaを扱えますが、非バックボーンarea間の経路は **area 0に接続するABR** を介して要約されます。DR選出・Stub/NSSA・仮想リンクは未実装です。ポートチャネルは静的設定のみ、VXLANのVTEP一覧は静的です。実機との差分も学習対象として、達成条件に書かれた観察範囲で取り組んでください。

## ⭐ Lv.0 — 操作に慣れる / 物理層 (L1)

### 課題 0-1: はじめての疎通（PC 2台を直結）
- **ねらい**: IPアドレス・サブネットマスクの基本、pingとは何か
- **使う機能**: PC ×2、結線、`set ip`、`ping`
- **手順のヒント**:
  1. パレットから **PC** を2台配置
  2. 「🔌 結線」で PC1 と PC2 を1本のケーブルで接続
  3. PC1 のコンソールで `set ip 10.0.0.1 255.255.255.0`
  4. PC2 のコンソールで `set ip 10.0.0.2 255.255.255.0`
  5. PC1 で `ping 10.0.0.2`
- **✅ 達成条件**: pingが応答を返す。パケットログに **ARP → ICMP echo/reply** が並ぶことを確認

### 課題 0-2: 同じネットワークとは？（ネットワーク部を変えて失敗させる）
- **ねらい**: IPアドレスとサブネットマスクから同一セグメントかどうかを判断する
- **使う機能**: 課題0-1の構成
- **手順のヒント**: PC2 を `set ip 10.0.1.2 255.255.255.0`（第3オクテットを変える）にして `ping` してみる
- **✅ 達成条件**: ルータが無い状態ではpingが **失敗**する。PC2を `10.0.0.x/24` に戻すと成功することを確認し、「同じ/異なるネットワーク」をマスクを使って説明できる

### 課題 0-3: リンクを落とす／上げる（shutdown / no shutdown）
- **ねらい**: 物理リンクのアップ/ダウン、`shutdown` の意味
- **使う機能**: 課題0-1の構成、ケーブルのリンク状態
- **手順のヒント**: 一方のケーブルを削除してから `ping`（失敗）→ 再結線して `ping`（成功）。ルータ/スイッチの場合はインターフェースを `shutdown` / `no shutdown` で試す
- **✅ 達成条件**: リンクダウン中は疎通せず、復旧すると疎通することを確認

### 課題 0-4: ハブは全員に配る（フラッディングを見る）
- **ねらい**: リピータハブは受信フレームを全ポートに中継する
- **使う機能**: ハブ ×1、PC ×3、`ping`、パケットログ
- **手順のヒント**: PC3台をハブに接続し全員同一セグメントに。PC1→PC2 に ping したとき、PC3 にもフレームが届いているかログで観察
- **✅ 達成条件**: 宛先でない PC3 にもフレームが流れている（＝ハブは宛先を見ない）ことをログで確認

### 課題 0-5: ARP を観察する
- **ねらい**: IP → MAC を解決する ARP の役割
- **使う機能**: `arp -a`、`ping`、パケットログ
- **手順のヒント**: `arp -a` で空を確認 → `ping` 実行 → 再度 `arp -a`
- **✅ 達成条件**: ping後にARPテーブルへ相手のMACが載る。ログで **ARP Request(ブロードキャスト)→ARP Reply(ユニキャスト)** を確認

## ⭐⭐ Lv.1 — スイッチとMAC / VLAN (L2)

### 課題 1-1: スイッチのMAC学習
- **ねらい**: L2スイッチはMACアドレステーブルを学習し、賢く転送する
- **使う機能**: L2スイッチ ×1、PC ×3、`show mac address-table`
- **手順のヒント**:
  1. スイッチにPC3台を接続し、PC1〜3をそれぞれ `10.0.0.1/24`、`10.0.0.2/24`、`10.0.0.3/24` に設定
  2. まだ通信していない状態でスイッチの `show mac address-table` を確認（空〜わずか）
  3. PC間で `ping` してから再度 `show mac address-table`
- **✅ 達成条件**: 通信したPCのMACがポートに紐づいて学習される

### 課題 1-2: ハブ vs スイッチ（転送範囲の違い）
- **ねらい**: 未学習/ブロードキャストはフラッディング、学習済みはユニキャスト転送
- **使う機能**: L2スイッチ、PC ×3、パケットログ
- **手順のヒント**: 新しい構成でPC1→PC2にpingする。最初に発生するARP RequestはブロードキャストなのでPC3にも届く。そのARP ReplyでMACを学習したあとに送られるICMP echoと、もう一度pingしたときのICMP echoを比較する。課題0-4のハブとも見比べる
- **✅ 達成条件**: ARPのブロードキャストはPC3にも届く一方、MAC学習後のICMPユニキャストは宛先ポートだけへ出ることをログで確認

### 課題 1-3: STP がL2ループを止める
- **ねらい**: 冗長L2リンクでSTPがループを防ぎ、どのポートを止めるかを理解する
- **使う機能**: L2スイッチ ×2、ケーブル ×2、`show spanning-tree`
- **手順のヒント**:
  1. スイッチ2台を **2本**のケーブルで接続し、両端は既定のVLAN 1のままにする
  2. 各スイッチで `show spanning-tree` を実行し、`root` / `designated` / `alternate` の役割と `forwarding` / `blocking` の状態を比べる
  3. PCを両側に1台ずつつなぎ、ARPまたはpingを発生させる。リンクを1本外してから戻し、表示が変わることも観察する
- **✅ 達成条件**: 冗長リンクの片側が `alternate / blocking` となり、通信は残る1本の転送リンクで成功することを確認

> この課題でブロードキャストストームやホップ上限の警告は期待しません。それは「STPが無いL2ループ」の現象ですが、本シミュレーターのスイッチではSTPが先に防止します。

### 課題 1-4: 静的ポートチャネルで束ねる
- **ねらい**: 冗長リンクを1本の論理リンクに束ね、帯域向上とループ回避
- **使う機能**: `channel-group <n> mode on`、`show etherchannel summary`
- **手順のヒント**: 課題1-3の2本のリンクを使う。**両スイッチの両方の接続ポート**で `channel-group 1 mode on` を設定し、2本とも同じVLANを運べるようにする。`show etherchannel summary` と `show spanning-tree` で状態を確認する
- **✅ 達成条件**: 2本が `Po1` として扱われ、STPで片方がblockingにならない。`show mac address-table` が `Po1` 表示となり、送信元・宛先・ポート番号を変えた複数フローで両メンバーに分散することをログで確認

### 課題 1-5: VLAN で分割する（同居しつつ隔離）
- **ねらい**: 1台のスイッチを2つの論理ネットワークに分ける
- **使う機能**: `switchport mode access`、`switchport access vlan <n>`
- **手順のヒント**:
  1. スイッチにPC4台。PC1/PC2 を VLAN10、PC3/PC4 を VLAN20 に割り当て
  2. 全PCを同じ `10.0.0.0/24` にしておく（あえて同一サブネット）
  3. 各アクセスポートに `switchport access vlan 10` / `20`
- **✅ 達成条件**: VLAN10同士・VLAN20同士は ping 成功、VLANを跨ぐと **失敗**（IPが同じでもL2で隔離される）

### 課題 1-6: トランクで VLAN をスイッチ間に延伸
- **ねらい**: 802.1Q タグ付きフレーム、トランクポート
- **使う機能**: L2スイッチ ×2、`switchport mode trunk`、`switchport trunk allowed vlan`
- **手順のヒント**: 2台のスイッチにそれぞれ VLAN10/20 のPCを配置。各PCの接続ポートをアクセスVLAN 10または20にし、スイッチ間リンクを両側 `switchport mode trunk` → `switchport trunk allowed vlan 10,20` にする。ログでVLANタグ付きフレームを確認
- **✅ 達成条件**: スイッチをまたいだ **同一VLAN** 同士が疎通。ログでトランク上のフレームにVLANタグが付くことを確認

### 課題 1-7: ネイティブVLAN の挙動
- **ねらい**: トランクでタグを付けない VLAN（ネイティブ）の意味
- **使う機能**: `switchport trunk native vlan <n>`
- **手順のヒント**: 課題1-6を土台に、VLAN10をトランクのnative VLANにする。両端を `switchport trunk native vlan 10` にした状態でVLAN10と20の通信を比較する。その後、片側だけnative VLANを20に変え、どちらのVLANが期待どおりに届かなくなるか観察してから設定を揃える
- **✅ 達成条件**: native VLAN（10）のフレームだけがトランク上でタグ無し、VLAN20はタグ付きで流れることを確認。native VLANの不一致ではタグ無しフレームが相手側で別VLANとして扱われることを説明できる

### 課題 1-8（応用）: STPルートブリッジを設計する
- **ねらい**: STPのルート選出と、冗長構成でどのリンクを転送させるかの設計
- **使う機能**: L2スイッチ ×3、三角形の結線、`spanning-tree vlan <n> priority <n>`、`show spanning-tree vlan <n>`
- **手順のヒント**:
  1. SW1〜SW3を三角形に結線し、全リンクをトランク、全PCをVLAN10にする
  2. 何も設定せずに各スイッチで `show spanning-tree vlan 10` を確認し、MACアドレスの大小で選ばれたルートを記録する
  3. SW1に `spanning-tree vlan 10 priority 24576` を設定し、SW1を必ずルートにする。どのポートがblockingへ変わるかを予想してから確認する
- **✅ 達成条件**: 優先度が低いSW1がルートになり、三角形のうち1経路だけがblockingとなる。任意の1本を外しても残りの経路でVLAN10の疎通が続くことを確認

## ⭐⭐⭐ Lv.2 — ルーティングの基本 (L3)

### 課題 2-1: ルータで2セグメントをつなぐ
- **ねらい**: デフォルトゲートウェイ、L3でセグメントをまたぐ
- **使う機能**: ルータ ×1、PC ×2、`ip address`、`no shutdown`、`set ip`（GW付き）
- **手順のヒント**:
  1. ルータの `Gi0/0` に `ip address 10.0.1.254 255.255.255.0` → `no shutdown`
  2. `Gi0/1` に `ip address 10.0.2.254 255.255.255.0` → `no shutdown`
  3. PC1: `set ip 10.0.1.10 255.255.255.0 10.0.1.254`
  4. PC2: `set ip 10.0.2.10 255.255.255.0 10.0.2.254`
  5. PC1 → PC2 に `ping`
- **✅ 達成条件**: 別セグメント間で ping 成功。`show ip route` に2つの `C`（直結）経路

### 課題 2-2: TTL とホップを見る（traceroute）
- **ねらい**: ルータを1つ越えるごとにTTLが減る
- **使う機能**: `traceroute`、パケットログ
- **手順のヒント**: 課題2-1で PC1 から `traceroute 10.0.2.10`。ログでIPヘッダのTTL値を追う
- **✅ 達成条件**: 経路にルータが1ホップ現れる。TTLがホップごとに減っていることを確認

### 課題 2-3: スタティックルートで3セグメント
- **ねらい**: 直結でない宛先へは経路を教える必要がある
- **使う機能**: ルータ ×2、`ip route <net> <mask> <nexthop>`
- **手順のヒント**:
  - 構成: `PC1 —(10.0.1.0/24)— RT1 —(10.0.99.0/30)— RT2 —(10.0.2.0/24)— PC2`
  - RT1: `ip route 10.0.2.0 255.255.255.0 10.0.99.2`
  - RT2: `ip route 10.0.1.0 255.255.255.0 10.0.99.1`
- **✅ 達成条件**: PC1 ↔ PC2 疎通。`show ip route` に `S`（スタティック）経路が載る。片方だけ設定して「戻りの経路が無い」失敗も体験する

### 課題 2-4: デフォルトルート（0.0.0.0/0）
- **ねらい**: 「その他すべて」をまとめて転送する経路
- **使う機能**: `ip route 0.0.0.0 0.0.0.0 <nexthop>`
- **手順のヒント**: 課題2-3で、RT1の「`10.0.2.0/24` 宛」経路だけを `ip route 0.0.0.0 0.0.0.0 10.0.99.2` に置き換える。RT2にはPC1のネットワークへの戻り経路を残す
- **✅ 達成条件**: RT1に個別経路が無くてもデフォルトルート経由で疎通し、`show ip route` に `S* 0.0.0.0/0` が表示される。戻り経路を消すと往復通信が失敗することも確認

### 課題 2-5: ロンゲストマッチ
- **ねらい**: 複数の経路が一致するとき、最も長い（詳しい）プレフィックスが優先される
- **使う機能**: `ip route`（範囲の広い経路 + 狭い経路の両方）
- **手順のヒント**:
  1. R1からR2（`172.16.12.0/30`）とR3（`172.16.13.0/30`）へ分岐させる。R2の先に `10.0.2.0/24`、R3の先に `10.0.3.0/24` のPCを置く（各ルータにはPC1のLANへの戻り経路も設定する）
  2. R1に `ip route 10.0.0.0 255.255.0.0 172.16.12.2` と `ip route 10.0.3.0 255.255.255.0 172.16.13.2` を設定する
  3. PC1から `10.0.2.x` と `10.0.3.x` へ `traceroute` する
- **✅ 達成条件**: `10.0.2.x` は広い `/16` 経路でR2へ、`10.0.3.x` はより長い `/24` 経路でR3へ向かうことを `show ip route` とtracerouteで確認

### 課題 2-6: L3スイッチ + SVI で VLAN間ルーティング
- **ねらい**: VLANをまたぐ通信を L3スイッチの SVI（interface vlan）で実現
- **使う機能**: L3スイッチ、`interface vlan <n>`、`ip address`
- **手順のヒント**:
  1. L3スイッチに VLAN10 / VLAN20 のアクセスポートとPCを用意
  2. `interface vlan 10` → `ip address 10.0.10.254 255.255.255.0`
  3. `interface vlan 20` → `ip address 10.0.20.254 255.255.255.0`
  4. 各PCのGWをそれぞれのSVIに
- **✅ 達成条件**: VLAN10 のPC と VLAN20 のPC が疎通（課題1-5では不可だったものがL3で可能に）

### 課題 2-7: DHCP でIPを自動配布
- **ねらい**: 手動設定なしでIP/GWを配る仕組み
- **使う機能**: `dhcp pool`、`set ip dhcp`、`dhcp leases`
- **手順のヒント**:
  1. サーバを同一LANに置き、まず `set ip 10.0.1.2 255.255.255.0 10.0.1.254` を設定する。ルータを置くならLAN側IFを `10.0.1.254/24` にして `no shutdown`
  2. サーバで `dhcp pool 10.0.1.0/24 10.0.1.100 10.0.1.199 10.0.1.254`
  3. クライアントPCで `set ip dhcp`。配布側で `dhcp leases`、クライアントで `ipconfig` を確認する
- **✅ 達成条件**: クライアントが自動でIPを取得。ログで **DISCOVER→OFFER→REQUEST→ACK** の流れを確認

### 課題 2-8: DHCPリレー（ip helper-address）
- **ねらい**: DHCPサーバが別セグメントにいる場合の中継
- **使う機能**: ルータ、`ip helper-address <server-ip>`
- **手順のヒント**:
  1. ルータのクライアント側を `10.0.10.254/24`、サーバ側を `10.0.20.254/24` にして、両IFを `no shutdown`。DHCPサーバは `10.0.20.2/24`、GWは `10.0.20.254` にする
  2. DHCPサーバに、**クライアント側ネットワーク用**の `dhcp pool 10.0.10.0/24 10.0.10.100 10.0.10.199 10.0.10.254` を設定する
  3. ルータの `10.0.10.0/24` 側IFへ `ip helper-address 10.0.20.2` を設定し、クライアントで `set ip dhcp` を実行する
- **✅ 達成条件**: 別セグメントのクライアントがDHCPでIPを取得する。クライアント側のDISCOVERはブロードキャストだが、リレー後はサーバへユニキャストで運ばれることをログで確認

### 課題 2-9（応用）: 3点で切り分ける経路障害
- **ねらい**: 「IP設定・デフォルトGW・往復の経路」のどこで通信が止まるかを、観測から特定する
- **使う機能**: 課題2-3の構成、`ipconfig`、`show ip interface brief`、`show ip route`、`traceroute`
- **手順のヒント**: 課題2-3が成功した状態を保存してから、次を1つずつ壊して原因を予想・記録する。(a) PC1のGWを誤った値にする、(b) RT1のPC2向け静的ルートを消す、(c) RT2のPC1向け戻りルートを消す。それぞれでPC1からping/traceroute、各ルータでルーティングテーブルを見る
- **✅ 達成条件**: 3種類の失敗について「最初に確認すべき表示」と「パケットが最後に到達する機器」を区別して説明でき、保存した正常構成へ戻して復旧できる

## ⭐⭐⭐⭐ Lv.3 — 動的ルーティングと冗長化 (L3)

### 課題 3-1: OSPF で経路を自動学習
- **ねらい**: 手動のスタティックルート無しで経路を交換する動的ルーティング
- **使う機能**: `router ospf <n>`、`network <net> <wild> area 0`、`show ip ospf neighbor`
- **手順のヒント**:
  1. ルータ2〜3台を直列/三角に接続し各IFにIP設定
  2. 各ルータで `router ospf 1` → `network 10.0.0.0 0.255.255.255 area 0`
  3. 速度を上げて収束を待つ → `show ip ospf neighbor` / `show ip route`
- **✅ 達成条件**: スタティック無しで端から端まで疎通。`show ip route` に `O`（OSPF）経路、`show ip ospf neighbor` に隣接

### 課題 3-2: OSPFコストで経路を操る
- **ねらい**: メトリック（コスト）による経路選択
- **使う機能**: `ip ospf cost <n>`
- **手順のヒント**: 三角トポロジで、優先させたくないリンクのコストを上げ、`show ip route` の経路が変わるのを確認
- **✅ 達成条件**: コスト変更前後で選択経路（via）が変わることを確認

### 課題 3-3: ECMP（等コスト複数経路）
- **ねらい**: 同コストの経路が複数あると負荷分散される
- **使う機能**: OSPF、複数の等コスト経路、`traceroute`
- **手順のヒント**: 2経路が等コストになるように構成。複数の送信元/宛先フローで `traceroute` を打つ
- **✅ 達成条件**: `show ip route` に同一宛先で複数 `via` 行。フローごとに別経路を通ることをログ/tracerouteで確認

### 課題 3-4: OSPF 再収束（障害時の自動迂回）
- **ねらい**: リンク障害時に別経路へ切り替わる
- **使う機能**: `shutdown`、`show ip route`
- **手順のヒント**: 冗長経路のある構成で、使用中リンクを `shutdown`。収束を待って再度 `ping` / `show ip route`
- **✅ 達成条件**: 一時的に途切れても、再収束後は別経路で疎通が回復する

### 課題 3-5: passive-interface
- **ねらい**: ホスト側IFにHelloを流さない設定（無駄・セキュリティ配慮）
- **使う機能**: `passive-interface <if>`
- **手順のヒント**: ホストが繋がるIFを `passive-interface` に。ルータ間IFはそのまま
- **✅ 達成条件**: 経路（サブネット）は広告されつつ、ホスト側IFにHelloが流れないことをログで確認

### 課題 3-6: VRRP で冗長ゲートウェイ
- **ねらい**: ゲートウェイの1台が落ちても通信を継続する仮想IP冗長
- **使う機能**: ルータ ×2、`vrrp <n> ip <vip>`、`show vrrp brief`
- **手順のヒント**:
  1. PC側LAN（`10.0.1.0/24`）とサーバ側LAN（`10.0.2.0/24`）を別々のL2スイッチで作り、RT1/RT2を両方のLANに接続する。各ルータの両IFへ個別IPを設定して `no shutdown`
  2. PC側ではRT1/RT2に `vrrp 1 ip 10.0.1.1`、サーバ側では `vrrp 2 ip 10.0.2.1` を設定する。PCのGWを `10.0.1.1`、サーバのGWを `10.0.2.1` にする
  3. `show vrrp brief` で各LANのMaster/Backupを確認してから、PC→サーバへ連続pingする
- **✅ 達成条件**: PCからサーバへの通信中にMaster側ルータの両IFを停止（`shutdown all` または各IFで`shutdown`）しても、一時的な途切れの後にBackupが引き継いで疎通する。ログで仮想MAC宛通信とgratuitous ARPを確認

### 課題 3-7: VRRP 優先度とプリエンプト
- **ねらい**: どちらを優先Masterにするかの制御
- **使う機能**: `vrrp <n> priority <n>`
- **手順のヒント**: RT1 を `priority 120`（デフォルト100）に。RT1復旧時にMasterへ戻るか確認
- **✅ 達成条件**: 優先度の高い方がMasterになり、復旧時にプリエンプトで奪い返すことを `show vrrp brief` で確認

### 課題 3-8（応用）: OSPFマルチareaとABR
- **ねらい**: area 0（バックボーン）、非バックボーンarea、ABRの役割と、area不一致で隣接しない理由を理解する
- **使う機能**: ルータ ×3、`network <net> <wild> area <area>`、`show ip ospf neighbor`、`show ip ospf database`
- **手順のヒント**:
  1. `PC-A — R1 — ABR — R2 — PC-B` を作る。R1—ABRを `10.12.0.0/24`、ABR—R2を `10.23.0.0/24`、PC-A側を `10.1.0.0/24`、PC-B側を `10.3.0.0/24` にする
  2. R1の参加IFをarea 1、R2の参加IFをarea 0にする。ABRは `network 10.12.0.0 0.0.0.255 area 1` と `network 10.23.0.0 0.0.0.255 area 0` を設定する。R1/R2のLAN側もそれぞれ同じareaで広告する
  3. 収束後、R1で `show ip ospf database` を確認し、ABR経由で受け取った `inter-area` の経路を探す。PC-AからPC-Bへtracerouteする
  4. R1側リンクのareaを一時的に0へ変更せず、ABR側だけarea 0へ変えた構成を作り、ネイバーが成立しないことを観察して戻す
- **✅ 達成条件**: R1とR2が直接隣接せず、ABRがarea 1とarea 0にそれぞれ隣接する。PC-A↔PC-Bが疎通し、`show ip ospf database` でABRが要約した `inter-area` の経路を確認できること、area不一致ではHelloが届いても隣接が成立しないことを確認

## ⭐⭐⭐⭐⭐ Lv.4 — トランスポート層とサービス (L4)

### 課題 4-1: TCP 3ウェイハンドシェイク
- **ねらい**: TCP接続確立（SYN / SYN-ACK / ACK）と終了（FIN/RST）
- **使う機能**: `http server on`、`http get`、パケットログ
- **手順のヒント**: サーバで `http server on`、クライアントで `http get <server-ip>`。ログでTCPフラグを追う
- **✅ 達成条件**: ログに **SYN → SYN,ACK → ACK**（確立）と、その後の FIN による終了が並ぶことを確認

### 課題 4-2: UDP を送る
- **ねらい**: コネクションレス通信、TCPとの違い
- **使う機能**: `udp listen <port>`、`udp send <ip> <port> <msg>`
- **手順のヒント**: 受信側 `udp listen 5000`、送信側 `udp send <ip> 5000 hello`
- **✅ 達成条件**: ハンドシェイク無しで単発のUDPが届く。TCP（課題4-1）とログを比較

### 課題 4-3: ACL で ICMP を止める
- **ねらい**: 拡張ACLによるフィルタリング
- **使う機能**: `access-list <num> deny icmp any host <ip>`、インターフェース適用
- **手順のヒント**: クライアントLANからサーバLANへ向かうルータの**クライアント側受信方向**に、次の順でACLを適用する。`access-list 100 deny icmp any host <server-ip>` → `access-list 100 permit ip any any` → 対象IFで `ip access-group 100 in`。ACLは先頭から評価され、最後に暗黙のdenyがある点に注意する
- **✅ 達成条件**: 対象への `ping` は失敗、`http get` は成功。`show access-lists` でルール順序を確認し、`permit ip any any` を外すとHTTPも失敗する理由を説明できる

### 課題 4-4: ACL で特定ポートだけ許可
- **ねらい**: プロトコル/ポート単位の制御（許可リスト）
- **使う機能**: `access-list <num> permit tcp any host <ip> eq 80`
- **手順のヒント**: クライアントLANからサーバへ入る方向に、`access-list 101 permit tcp any host <server-ip> eq 80` のみを設定して `ip access-group 101 in` を適用する。明示的な許可以外は暗黙に拒否される。サーバでは `http server on` と `udp listen 5000` を実行する
- **✅ 達成条件**: `http get <server-ip>`（80番）は成功し、`udp send <server-ip> 5000 hello` は遮断される。反対方向にACLを付けた場合と、クライアント側受信方向に付けた場合で結果が違うことを観察する

### 課題 4-5: L4ロードバランサ（ラウンドロビン）
- **ねらい**: 複数バックエンドへのTCP分散
- **使う機能**: LB、`lb service 80`、`lb backend add <ip>`、`lb status`
- **手順のヒント**:
  1. バックエンドのサーバ2〜3台で `http server on`
  2. LBで `lb service 80` → 各バックエンドを `lb backend add <ip>`
  3. クライアントから LB のVIPへ複数回 `http get`
- **✅ 達成条件**: リクエストが各バックエンドに順番に振り分けられることを `lb status` / ログで確認

### 課題 4-6: ヘルスチェックとフェイルアウト
- **ねらい**: 死んだバックエンドを自動的に振り分けから外す
- **使う機能**: `http server off`、`lb status`
- **手順のヒント**: 課題4-5の状態で、1台のバックエンドを `http server off`。しばらく待ってから再度アクセス
- **✅ 達成条件**: `lb status` で該当バックエンドが **DOWN** 判定になり、以降そのサーバには振り分けられない。復旧（`http server on`）で自動復帰

### 課題 4-7（応用）: PATで複数クライアントを1つのグローバルIPへ集約
- **ねらい**: NAT/PATのinside local / inside globalと、ポート番号による多重化
- **使う機能**: ルータ、`ip nat inside|outside`、`ip nat inside source list ... overload`、`show ip nat translations`
- **手順のヒント**:
  1. 内部LAN（例: `10.0.1.0/24`）—NATルータ—外部LAN（例: `203.0.113.0/24`）を作り、外部側にHTTPサーバを置く。ルータの外部IFは `203.0.113.1` とする
  2. 内部IFで `ip nat inside`、外部IFで `ip nat outside`。次に `access-list 100 permit ip 10.0.1.0 0.0.0.255 any` と `ip nat inside source list 100 interface Gi0/1 overload` を設定する（IF名は外部側に合わせる）
  3. 異なる内部PCから外部HTTPサーバへアクセスし、ルータで `show ip nat translations` を見る
- **✅ 達成条件**: 複数の内部IPが同じinside global IPへ変換され、TCPの送信元ポートで区別されることを確認。ACL対象外の送信元は変換されないことも試す

## 🏆 Lv.5 — 大規模・総合設計 (L1〜L4)

### 課題 5-1: スパイン・リーフを生成して理解する
- **ねらい**: データセンターの主流トポロジ、L3-to-the-leaf
- **使う機能**: 「🏭 DC生成」、`show ip route`、`traceroute`
- **手順のヒント**: DC生成でスパイン2 × リーフ4 × ホスト12 程度を作成。ホスト間で `traceroute`
- **✅ 達成条件**: 任意のホスト間が疎通。リーフ↔スパイン間が /30・OSPF+ECMPで自動構成されていることを確認

### 課題 5-2: ECMP をファブリックで観察
- **ねらい**: 複数スパインへの負荷分散
- **使う機能**: `traceroute`（複数フロー）
- **手順のヒント**: 異なる送信元/宛先の組み合わせで `traceroute` を打ち、通過スパインが分かれるか観察
- **✅ 達成条件**: フローによって別のスパインを経由する（ECMPハッシュ分散）ことを確認

### 課題 5-3: スパイン障害からの再収束
- **ねらい**: 大規模環境での耐障害性
- **使う機能**: スパインの `shutdown`、OSPF再収束
- **手順のヒント**: スパインを1台 `shutdown`。収束後に全ホスト疎通を再確認
- **✅ 達成条件**: 残りのスパイン経由で全ホストの疎通が維持される

### 課題 5-4: 大規模生成とスケール体験
- **ねらい**: エンジンのスケールと一括操作
- **使う機能**: DC生成（大きめ）、Shift+ドラッグ複数選択、一括CLI/一括ping、🔍検索、⛶全体表示、🧭整列
- **手順のヒント**: ホスト数を増やして生成 → 複数選択で一括ping → 検索やレイアウトで大規模構成を扱う練習
- **✅ 達成条件**: 200ホスト規模でも生成→収束→全ホストping が短時間で完了することを体感

### 課題 5-5【総合】ゼロから作るミニデータセンター
- **ねらい**: これまでの技術を組み合わせて一つのネットワークを設計・構築する集大成
- **使う機能**: VLAN + トランク + SVI(VLAN間ルーティング) + DHCP + OSPF + VRRP + ACL + LB
- **要件（設計してみよう）**:
  1. **アクセス層**: L2スイッチ2台に、VLAN10（サーバ）とVLAN20（クライアント）を用意し、トランクで接続
  2. **配布**: クライアントVLANは **DHCP** で自動配布（サーバは別セグメント→ `ip helper-address` リレー）
  3. **ゲートウェイ冗長**: ルータ or L3スイッチ2台で各VLANのGWを **VRRP** 冗長化
  4. **コア**: コア間は **OSPF** で動的ルーティング（可能なら **ECMP** で2経路）
  5. **サービス**: サーバVLANに **LB + 複数バックエンド**（HTTP）を配置し、クライアントからLB経由でアクセス
  6. **セキュリティ**: クライアントからサーバへの **不要なICMPをACLで遮断**、HTTP(80)は許可
- **✅ 達成条件（すべて満たす）**:
  - [ ] クライアントがDHCPでIP取得できる
  - [ ] VLAN間・セグメント間がOSPF/SVIで疎通する
  - [ ] GWを1台落としてもVRRPで通信が継続する
  - [ ] LB経由のHTTPが複数バックエンドに分散され、1台落とすと自動迂回する
  - [ ] ACLでICMPが遮断され、HTTPは通る

### 課題 5-6（応用）: 障害対応ランブックを作る
- **ねらい**: 正常系の構築だけでなく、障害を観測して安全に復旧する運用手順を身につける
- **使う機能**: 課題5-5の構成、`show ip route`、`show ip ospf neighbor`、`show vrrp brief`、`show etherchannel summary`、`lb status`、パケットログ、保存/書出
- **手順のヒント**:
  1. 課題5-5の正常構成を保存し、クライアントからHTTPとpingを実行したときの「正常時の表示」を記録する
  2. 次を1件ずつ発生させ、各障害で「影響・最初の確認コマンド・復旧確認」を表にする。(a) OSPFの冗長リンク停止、(b) VRRP Masterの停止、(c) ポートチャネルの片メンバー停止、(d) HTTPバックエンド停止、(e) ACLの誤適用
  3. 同時に2件は起こさず、保存した正常構成へ戻してから次を試す
- **✅ 達成条件**: 少なくとも5種類の障害について、観測結果に基づく一次切り分け手順と復旧後の確認項目を自分の言葉でまとめられる

## 🧭 Lv.6 — 実務に近い境界・オーバーレイ設計 (L2〜L4)

### 課題 6-1: 静的NATで内部Webサーバを公開する
- **ねらい**: 外部公開用アドレスと内部アドレスを1対1で対応付け、PATとの用途の違いを理解する
- **使う機能**: ルータ、`ip nat inside|outside`、`ip nat inside source static`、`show ip nat translations`
- **手順のヒント**:
  1. 課題4-7のNATルータを使い、内部にWebサーバ（例: `10.0.1.50`）、外部にクライアントを置く。内部IFをinside、外部IFをoutsideに設定する
  2. `ip nat inside source static 10.0.1.50 203.0.113.50` を設定し、外部クライアントから `http get 203.0.113.50` を実行する
  3. 内部クライアントの外向きPATと同時に試し、`show ip nat translations` を比較する
- **✅ 達成条件**: 外部クライアントはグローバルIPへ接続しているのに、内部Webサーバへ到達することを確認。静的NATの固定行とPATの動的なポート付き行を区別して説明できる

### 課題 6-2: VXLANで離れたVLANを延伸する
- **ねらい**: L3 underlay上でL2 EthernetフレームをUDP/4789にカプセル化する理由と、VNIの役割
- **使う機能**: 「サンプル6: VXLAN tenant overlay」、`show vxlan`、パケットログ
- **手順のヒント**:
  1. サンプル6を開き、各VTEPの `show vxlan` と `show ip route` で「VNI/VLAN対応」と「相手VTEPへのIP経路」を確認する
  2. `CLIENT1` から遠隔サイトの `10.244.10.23` へ `ping`、続いて `http get 10.244.10.23` を実行する
  3. パケットログで、inner Ethernetフレームが外側のIP/UDP（宛先ポート4789）に包まれる記録を探す。初回と2回目でMAC-to-VTEP学習後の転送も比べる
- **✅ 達成条件**: 同じVLAN10にいる遠隔ホストがL3 underlay越しに通信でき、ログでUDP/4789とVNI 10100を確認。VXLANを削除した場合に何が届かなくなるか説明できる

### 課題 6-3（発展）: テナント分離の設計レビュー
- **ねらい**: VLAN/VNI、ACL、NATを役割ごとに使い分け、要件から最小構成を選ぶ
- **使う機能**: VLAN、SVI、ACL、NAT/PAT、VXLAN（必要なものだけ）
- **要件（設計してみよう）**:
  1. Tenant AとTenant Bは、同じ物理スイッチ群を共有しても相互通信できない
  2. Tenant Aだけが外部HTTPへ出られる。外部からはTenant AのWebサーバだけを公開する
  3. Tenant Aの2拠点はL3網をまたいでも同一L2セグメントとして扱いたい
  4. Tenant Bの内部管理通信は外部へ出してはいけない
- **手順のヒント**: まず「L2分離（VLAN/VNI）」「L3境界（SVI/ルータ）」「通してよい通信（ACL）」「アドレス変換（NAT）」を4列の表にして設計する。構築後は、許可される通信と拒否される通信を最低4ケースずつ試す。VXLANは静的VTEP peer方式であることを前提にする
- **✅ 達成条件**: 要件1〜4をすべて試験で示し、各技術を採用した理由と、VLANだけ・ACLだけでは満たせない要件を説明できる

### 課題 6-4（発展）: Kubernetes風マルチホスト・オーバーレイ
- **ねらい**: 物理ホスト、ToR、Podネットワーク、Service、外部公開の関係を一つの構成で追う
- **使う機能**: [Kubernetesマッピング](k8s-mapping.md)、[マルチホスト構成](k8s-multihost.md)、VXLAN、OSPF、NAT/PAT、LB
- **手順のヒント**: `docs/k8s-multihost.md` の構成を段階的に作る。まずunderlayのOSPF疎通、次に各ノードのPod網、Service LB、最後に境界ルータのNATを追加する。各段階で保存し、同一物理ホスト内・別物理ホスト間・外部クライアントからの3方向で経路を比較する
- **✅ 達成条件**: Pod間、Service経由、外部→公開サービス、Pod→外部の4通信を成功させる。さらにHostSWを1台停止したとき、どのノード群が同時に影響を受けるかをパケットログと構成図から説明できる

## 学習の進め方（おすすめルート）

```
Lv.0（物理とping）
   └─▶ Lv.1（L2: スイッチ→VLAN→トランク→静的ポートチャネル）
          └─▶ Lv.2（L3: ルータ→スタティック→SVI→DHCP）
                 └─▶ Lv.3（OSPF→ECMP→VRRP）
                        └─▶ Lv.4（TCP/UDP→ACL→LB）
                               └─▶ Lv.5（ファブリック→総合設計→障害対応）
                                      └─▶ Lv.6（NAT→VXLAN→テナント/マルチホスト設計）
```

- つまずいたら **パケットログ** を開いて「今どのフレームがどこを流れているか」を必ず目で確認しましょう
- 「うまくいかない」課題（0-2 / 2-3の片側設定 など）は、**わざと失敗して原因を切り分ける**練習です
- 余裕があれば、各課題を `💾 保存` / `⬇ 書出` で残し、あとから比較・再現できるようにしておくと理解が深まります
