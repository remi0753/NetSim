# NetSim 課題集 — ネットワーク初学者のための構築チャレンジ

このシミュレーター（NetSim）で実際に手を動かしながら、L1〜L4 のネットワークを
段階的に学ぶための課題集です。難易度ごとに整理してあり、上から順にこなすと
「ケーブル1本」から「ミニデータセンター」まで自然にステップアップできます。

各課題は以下の形式で書かれています。

- **ねらい** — この課題で身につく概念
- **使う機能** — 必要なデバイス・コマンド
- **手順のヒント** — 詰まったときの道しるべ
- **✅ 達成条件** — 「できた」と判断するための確認方法

> 💡 全課題共通のコツ
> - デバイスを **ダブルクリック** するとコンソール（CLI）が開きます
> - 下部の **パケットログ** は行をクリックするとレイヤ別にデコード表示されます
> - 速度スライダ（x0.25〜x64）で、OSPF収束などは早送りすると快適です
> - ルータの新品インターフェースは `shutdown` 状態です。`no shutdown` を忘れずに

## ⭐ Lv.0 — 操作に慣れる / 物理層 (L1)

### 課題 0-1: はじめての疎通（PC 2台を直結）
- **ねらい**: IPアドレス・サブネットマスクの基本、pingとは何か
- **使う機能**: PC ×2、結線、`set ip`、`ping`
- **手順のヒント**:
  1. パレットから **PC** を2台配置
  2. 「🔌 結線」で PC1 と PC2 を1本のケーブルで接続
  3. PC1 のコンソールで `set ip 10.0.0.1 255.255.255.0`
  4. PC2 のコンソールで `set ip 10.0.0.2 255.255.255.0`
  5. PC1 で `ping 10.0.0.2`
- **✅ 達成条件**: pingが応答を返す。パケットログに **ARP → ICMP echo/reply** が並ぶことを確認

### 課題 0-2: 同じネットワークとは？（マスク違いで失敗させる）
- **ねらい**: サブネットマスクが違うと通信できない理由を体感する
- **使う機能**: 課題0-1の構成
- **手順のヒント**: PC2 を `set ip 10.0.1.2 255.255.255.0`（第3オクテットを変える）にして `ping` してみる
- **✅ 達成条件**: pingが **失敗**する。マスクを揃える or 同一セグメントのIPに直すと成功する、を両方確認

### 課題 0-3: リンクを落とす／上げる（shutdown / no shutdown）
- **ねらい**: 物理リンクのアップ/ダウン、`shutdown` の意味
- **使う機能**: 課題0-1の構成、ケーブルのリンク状態
- **手順のヒント**: 一方のケーブルを削除してから `ping`（失敗）→ 再結線して `ping`（成功）。ルータ/スイッチの場合はインターフェースを `shutdown` / `no shutdown` で試す
- **✅ 達成条件**: リンクダウン中は疎通せず、復旧すると疎通することを確認

### 課題 0-4: ハブは全員に配る（フラッディングを見る）
- **ねらい**: リピータハブは受信フレームを全ポートに中継する（衝突ドメイン）
- **使う機能**: ハブ ×1、PC ×3、`ping`、パケットログ
- **手順のヒント**: PC3台をハブに接続し全員同一セグメントに。PC1→PC2 に ping したとき、PC3 にもフレームが届いているかログで観察
- **✅ 達成条件**: 宛先でない PC3 にもフレームが流れている（＝ハブは宛先を見ない）ことをログで確認

### 課題 0-5: ARP を観察する
- **ねらい**: IP → MAC を解決する ARP の役割
- **使う機能**: `arp -a`、`ping`、パケットログ
- **手順のヒント**: `arp -a` で空を確認 → `ping` 実行 → 再度 `arp -a`
- **✅ 達成条件**: ping後にARPテーブルへ相手のMACが載る。ログで **ARP Request(ブロードキャスト)→ARP Reply(ユニキャスト)** を確認

## ⭐⭐ Lv.1 — スイッチとMAC / VLAN (L2)

### 課題 1-1: スイッチのMAC学習
- **ねらい**: L2スイッチはMACアドレステーブルを学習し、賢く転送する
- **使う機能**: L2スイッチ ×1、PC ×3、`show mac address-table`
- **手順のヒント**:
  1. スイッチにPC3台を接続し、全PCを `10.0.0.0/24` で設定
  2. まだ通信していない状態でスイッチの `show mac address-table` を確認（空〜わずか）
  3. PC間で `ping` してから再度 `show mac address-table`
- **✅ 達成条件**: 通信したPCのMACがポートに紐づいて学習される

### 課題 1-2: ハブ vs スイッチ（転送範囲の違い）
- **ねらい**: 未学習/ブロードキャストはフラッディング、学習済みはユニキャスト転送
- **使う機能**: L2スイッチ、PC ×3、パケットログ
- **手順のヒント**: 1回目のpingではフラッディング、2回目以降は該当ポートだけに転送されることをログで比較。課題0-4のハブと見比べる
- **✅ 達成条件**: 学習後は宛先ポート以外にフレームが出ないことをログで確認

### 課題 1-3: ループとブロードキャストストーム
- **ねらい**: L2ループの危険性（STPが無い環境での挙動）
- **使う機能**: L2スイッチ ×2、ケーブル ×2
- **手順のヒント**: スイッチ2台を **2本**のケーブルで接続。ブロードキャスト（ARP等）を発生させる
- **✅ 達成条件**: ループ検知の警告（ホップ上限）を確認。次の課題1-4で解消する

### 課題 1-4: 静的ポートチャネルで束ねる
- **ねらい**: 冗長リンクを1本の論理リンクに束ね、帯域向上とループ回避
- **使う機能**: `channel-group <n> mode on`、`show etherchannel summary`
- **手順のヒント**: 課題1-3の2本のリンク両端で、各インターフェースに `channel-group 1 mode on`。`show etherchannel summary` で `Po1` を確認
- **✅ 達成条件**: ループ警告が消え、`show mac address-table` でポートが `Po1` 表示になる。複数フローで両リンクに分散されることをログで確認

### 課題 1-5: VLAN で分割する（同居しつつ隔離）
- **ねらい**: 1台のスイッチを2つの論理ネットワークに分ける
- **使う機能**: `switchport mode access`、`switchport access vlan <n>`
- **手順のヒント**:
  1. スイッチにPC4台。PC1/PC2 を VLAN10、PC3/PC4 を VLAN20 に割り当て
  2. 全PCを同じ `10.0.0.0/24` にしておく（あえて同一サブネット）
  3. 各アクセスポートに `switchport access vlan 10` / `20`
- **✅ 達成条件**: VLAN10同士・VLAN20同士は ping 成功、VLANを跨ぐと **失敗**（IPが同じでもL2で隔離される）

### 課題 1-6: トランクで VLAN をスイッチ間に延伸
- **ねらい**: 802.1Q タグ付きフレーム、トランクポート
- **使う機能**: L2スイッチ ×2、`switchport mode trunk`、`switchport trunk allowed vlan`
- **手順のヒント**: 2台のスイッチにそれぞれ VLAN10/20 のPCを配置。スイッチ間リンクを両側 `switchport mode trunk`。ログでVLANタグ付きフレームを確認
- **✅ 達成条件**: スイッチをまたいだ **同一VLAN** 同士が疎通。ログでトランク上のフレームにVLANタグが付くことを確認

### 課題 1-7: ネイティブVLAN の挙動
- **ねらい**: トランクでタグを付けない VLAN（ネイティブ）の意味
- **使う機能**: `switchport trunk native vlan <n>`
- **手順のヒント**: トランクの片側で native vlan を変え、ミスマッチ時の挙動を観察 → 揃える
- **✅ 達成条件**: ネイティブVLANのフレームがタグ無しで流れることをログで確認。両端の設定を揃えると期待どおり疎通

## ⭐⭐⭐ Lv.2 — ルーティングの基本 (L3)

### 課題 2-1: ルータで2セグメントをつなぐ
- **ねらい**: デフォルトゲートウェイ、L3でセグメントをまたぐ
- **使う機能**: ルータ ×1、PC ×2、`ip address`、`no shutdown`、`set ip`（GW付き）
- **手順のヒント**:
  1. ルータの `Gi0/0` に `ip address 10.0.1.254 255.255.255.0` → `no shutdown`
  2. `Gi0/1` に `ip address 10.0.2.254 255.255.255.0` → `no shutdown`
  3. PC1: `set ip 10.0.1.10 255.255.255.0 10.0.1.254`
  4. PC2: `set ip 10.0.2.10 255.255.255.0 10.0.2.254`
  5. PC1 → PC2 に `ping`
- **✅ 達成条件**: 別セグメント間で ping 成功。`show ip route` に2つの `C`（直結）経路

### 課題 2-2: TTL とホップを見る（traceroute）
- **ねらい**: ルータを1つ越えるごとにTTLが減る
- **使う機能**: `traceroute`、パケットログ
- **手順のヒント**: 課題2-1で PC1 から `traceroute 10.0.2.10`。ログでIPヘッダのTTL値を追う
- **✅ 達成条件**: 経路にルータが1ホップ現れる。TTLがホップごとに減っていることを確認

### 課題 2-3: スタティックルートで3セグメント
- **ねらい**: 直結でない宛先へは経路を教える必要がある
- **使う機能**: ルータ ×2、`ip route <net> <mask> <nexthop>`
- **手順のヒント**:
  - 構成: `PC1 —(10.0.1.0/24)— RT1 —(10.0.99.0/30)— RT2 —(10.0.2.0/24)— PC2`
  - RT1: `ip route 10.0.2.0 255.255.255.0 10.0.99.2`
  - RT2: `ip route 10.0.1.0 255.255.255.0 10.0.99.1`
- **✅ 達成条件**: PC1 ↔ PC2 疎通。`show ip route` に `S`（スタティック）経路が載る。片方だけ設定して「戻りの経路が無い」失敗も体験する

### 課題 2-4: デフォルトルート（0.0.0.0/0）
- **ねらい**: 「その他すべて」をまとめて転送する経路
- **使う機能**: `ip route 0.0.0.0 0.0.0.0 <nexthop>`
- **手順のヒント**: 課題2-3のスタティックルートをデフォルトルートに置き換えてみる
- **✅ 達成条件**: 個別経路が無くてもデフォルトルート経由で疎通。`show ip route` に `S* 0.0.0.0/0`

### 課題 2-5: ロンゲストマッチ
- **ねらい**: 複数の経路が一致するとき、最も長い（詳しい）プレフィックスが優先される
- **使う機能**: `ip route`（範囲の広い経路 + 狭い経路の両方）
- **手順のヒント**: `10.0.0.0/16` と `10.0.2.0/24` を別ネクストホップで設定し、`10.0.2.10` 宛の経路選択を観察
- **✅ 達成条件**: 宛先が `/24` にマッチする場合はそちらが選ばれることを `show ip route` と実際の経路で確認

### 課題 2-6: L3スイッチ + SVI で VLAN間ルーティング
- **ねらい**: VLANをまたぐ通信を L3スイッチの SVI（interface vlan）で実現
- **使う機能**: L3スイッチ、`interface vlan <n>`、`ip address`
- **手順のヒント**:
  1. L3スイッチに VLAN10 / VLAN20 のアクセスポートとPCを用意
  2. `interface vlan 10` → `ip address 10.0.10.254 255.255.255.0`
  3. `interface vlan 20` → `ip address 10.0.20.254 255.255.255.0`
  4. 各PCのGWをそれぞれのSVIに
- **✅ 達成条件**: VLAN10 のPC と VLAN20 のPC が疎通（課題1-5では不可だったものがL3で可能に）

### 課題 2-7: DHCP でIPを自動配布
- **ねらい**: 手動設定なしでIP/GWを配る仕組み
- **使う機能**: `dhcp pool`、`set ip dhcp`、`dhcp leases`
- **手順のヒント**:
  1. サーバ（またはPC）で `dhcp pool 10.0.1.0/24 10.0.1.100 10.0.1.199 10.0.1.254`
  2. クライアントPCで `set ip dhcp`
  3. 配布側で `dhcp leases`、クライアントで `ipconfig`
- **✅ 達成条件**: クライアントが自動でIPを取得。ログで **DISCOVER→OFFER→REQUEST→ACK** の流れを確認

### 課題 2-8: DHCPリレー（ip helper-address）
- **ねらい**: DHCPサーバが別セグメントにいる場合の中継
- **使う機能**: ルータ、`ip helper-address <server-ip>`
- **手順のヒント**: クライアント側セグメントのルータIFに `ip helper-address <DHCPサーバのIP>`。サーバは別セグメントに配置
- **✅ 達成条件**: 別セグメントのクライアントがDHCPでIP取得。ブロードキャストがユニキャストで中継される様子をログで確認

## ⭐⭐⭐⭐ Lv.3 — 動的ルーティングと冗長化 (L3)

### 課題 3-1: OSPF で経路を自動学習
- **ねらい**: 手動のスタティックルート無しで経路を交換する動的ルーティング
- **使う機能**: `router ospf <n>`、`network <net> <wild> area 0`、`show ip ospf neighbor`
- **手順のヒント**:
  1. ルータ2〜3台を直列/三角に接続し各IFにIP設定
  2. 各ルータで `router ospf 1` → `network 10.0.0.0 0.255.255.255 area 0`
  3. 速度を上げて収束を待つ → `show ip ospf neighbor` / `show ip route`
- **✅ 達成条件**: スタティック無しで端から端まで疎通。`show ip route` に `O`（OSPF）経路、`show ip ospf neighbor` に隣接

### 課題 3-2: OSPFコストで経路を操る
- **ねらい**: メトリック（コスト）による経路選択
- **使う機能**: `ip ospf cost <n>`
- **手順のヒント**: 三角トポロジで、優先させたくないリンクのコストを上げ、`show ip route` の経路が変わるのを確認
- **✅ 達成条件**: コスト変更前後で選択経路（via）が変わることを確認

### 課題 3-3: ECMP（等コスト複数経路）
- **ねらい**: 同コストの経路が複数あると負荷分散される
- **使う機能**: OSPF、複数の等コスト経路、`traceroute`
- **手順のヒント**: 2経路が等コストになるように構成。複数の送信元/宛先フローで `traceroute` を打つ
- **✅ 達成条件**: `show ip route` に同一宛先で複数 `via` 行。フローごとに別経路を通ることをログ/tracerouteで確認

### 課題 3-4: OSPF 再収束（障害時の自動迂回）
- **ねらい**: リンク障害時に別経路へ切り替わる
- **使う機能**: `shutdown`、`show ip route`
- **手順のヒント**: 冗長経路のある構成で、使用中リンクを `shutdown`。収束を待って再度 `ping` / `show ip route`
- **✅ 達成条件**: 一時的に途切れても、再収束後は別経路で疎通が回復する

### 課題 3-5: passive-interface
- **ねらい**: ホスト側IFにHelloを流さない設定（無駄・セキュリティ配慮）
- **使う機能**: `passive-interface <if>`
- **手順のヒント**: ホストが繋がるIFを `passive-interface` に。ルータ間IFはそのまま
- **✅ 達成条件**: 経路（サブネット）は広告されつつ、ホスト側IFにHelloが流れないことをログで確認

### 課題 3-6: VRRP で冗長ゲートウェイ
- **ねらい**: ゲートウェイの1台が落ちても通信を継続する仮想IP冗長
- **使う機能**: ルータ ×2、`vrrp <n> ip <vip>`、`show vrrp brief`
- **手順のヒント**:
  1. RT1/RT2 を同一セグメントに接続、両方 `vrrp 1 ip 10.0.1.1`
  2. PCのGWは仮想IP `10.0.1.1`
  3. `show vrrp brief` で Master/Backup を確認
- **✅ 達成条件**: 継続 ping しながら Master 側IFを `shutdown` → 一瞬途切れて Backup が引き継ぐ。gratuitous ARP の流れをログで確認

### 課題 3-7: VRRP 優先度とプリエンプト
- **ねらい**: どちらを優先Masterにするかの制御
- **使う機能**: `vrrp <n> priority <n>`
- **手順のヒント**: RT1 を `priority 120`（デフォルト100）に。RT1復旧時にMasterへ戻るか確認
- **✅ 達成条件**: 優先度の高い方がMasterになり、復旧時にプリエンプトで奪い返すことを `show vrrp brief` で確認

## ⭐⭐⭐⭐⭐ Lv.4 — トランスポート層とサービス (L4)

### 課題 4-1: TCP 3ウェイハンドシェイク
- **ねらい**: TCP接続確立（SYN / SYN-ACK / ACK）と終了（FIN/RST）
- **使う機能**: `http server on`、`http get`、パケットログ
- **手順のヒント**: サーバで `http server on`、クライアントで `http get <server-ip>`。ログでTCPフラグを追う
- **✅ 達成条件**: ログに **SYN → SYN,ACK → ACK**（確立）と、その後の FIN による終了が並ぶことを確認

### 課題 4-2: UDP を送る
- **ねらい**: コネクションレス通信、TCPとの違い
- **使う機能**: `udp listen <port>`、`udp send <ip> <port> <msg>`
- **手順のヒント**: 受信側 `udp listen 5000`、送信側 `udp send <ip> 5000 hello`
- **✅ 達成条件**: ハンドシェイク無しで単発のUDPが届く。TCP（課題4-1）とログを比較

### 課題 4-3: ACL で ICMP を止める
- **ねらい**: 拡張ACLによるフィルタリング
- **使う機能**: `access-list <num> deny icmp any host <ip>`、インターフェース適用
- **手順のヒント**: あるサーバ宛のICMPを拒否するACLを作り、pingが止まることを確認。他の通信（HTTP等）は通す
- **✅ 達成条件**: 対象への `ping` は失敗、`http get` は成功。ACLヒットの様子を確認（`show access-lists`）

### 課題 4-4: ACL で特定ポートだけ許可
- **ねらい**: プロトコル/ポート単位の制御（許可リスト）
- **使う機能**: `access-list <num> permit tcp any host <ip> eq 80`
- **手順のヒント**: HTTP(80)は許可し、それ以外のTCP/UDPを拒否する構成
- **✅ 達成条件**: `http get`（80番）は成功、別ポートのUDP等は遮断されることを確認

### 課題 4-5: L4ロードバランサ（ラウンドロビン）
- **ねらい**: 複数バックエンドへのTCP分散
- **使う機能**: LB、`lb service 80`、`lb backend add <ip>`、`lb status`
- **手順のヒント**:
  1. バックエンドのサーバ2〜3台で `http server on`
  2. LBで `lb service 80` → 各バックエンドを `lb backend add <ip>`
  3. クライアントから LB のVIPへ複数回 `http get`
- **✅ 達成条件**: リクエストが各バックエンドに順番に振り分けられることを `lb status` / ログで確認

### 課題 4-6: ヘルスチェックとフェイルアウト
- **ねらい**: 死んだバックエンドを自動的に振り分けから外す
- **使う機能**: `http server off`、`lb status`
- **手順のヒント**: 課題4-5の状態で、1台のバックエンドを `http server off`。しばらく待ってから再度アクセス
- **✅ 達成条件**: `lb status` で該当バックエンドが **DOWN** 判定になり、以降そのサーバには振り分けられない。復旧（`http server on`）で自動復帰

## 🏆 Lv.5 — 大規模・総合設計 (L1〜L4)

### 課題 5-1: スパイン・リーフを生成して理解する
- **ねらい**: データセンターの主流トポロジ、L3-to-the-leaf
- **使う機能**: 「🏭 DC生成」、`show ip route`、`traceroute`
- **手順のヒント**: DC生成でスパイン2 × リーフ4 × ホスト12 程度を作成。ホスト間で `traceroute`
- **✅ 達成条件**: 任意のホスト間が疎通。リーフ↔スパイン間が /30・OSPF+ECMPで自動構成されていることを確認

### 課題 5-2: ECMP をファブリックで観察
- **ねらい**: 複数スパインへの負荷分散
- **使う機能**: `traceroute`（複数フロー）
- **手順のヒント**: 異なる送信元/宛先の組み合わせで `traceroute` を打ち、通過スパインが分かれるか観察
- **✅ 達成条件**: フローによって別のスパインを経由する（ECMPハッシュ分散）ことを確認

### 課題 5-3: スパイン障害からの再収束
- **ねらい**: 大規模環境での耐障害性
- **使う機能**: スパインの `shutdown`、OSPF再収束
- **手順のヒント**: スパインを1台 `shutdown`。収束後に全ホスト疎通を再確認
- **✅ 達成条件**: 残りのスパイン経由で全ホストの疎通が維持される

### 課題 5-4: 大規模生成とスケール体験
- **ねらい**: エンジンのスケールと一括操作
- **使う機能**: DC生成（大きめ）、Shift+ドラッグ複数選択、一括CLI/一括ping、🔍検索、⛶全体表示、🧭整列
- **手順のヒント**: ホスト数を増やして生成 → 複数選択で一括ping → 検索やレイアウトで大規模構成を扱う練習
- **✅ 達成条件**: 200ホスト規模でも生成→収束→全ホストping が短時間で完了することを体感

### 課題 5-5【総合】ゼロから作るミニデータセンター
- **ねらい**: これまでの技術を組み合わせて一つのネットワークを設計・構築する集大成
- **使う機能**: VLAN + トランク + SVI(VLAN間ルーティング) + DHCP + OSPF + VRRP + ACL + LB
- **要件（設計してみよう）**:
  1. **アクセス層**: L2スイッチ2台に、VLAN10（サーバ）とVLAN20（クライアント）を用意し、トランクで接続
  2. **配布**: クライアントVLANは **DHCP** で自動配布（サーバは別セグメント→ `ip helper-address` リレー）
  3. **ゲートウェイ冗長**: ルータ or L3スイッチ2台で各VLANのGWを **VRRP** 冗長化
  4. **コア**: コア間は **OSPF** で動的ルーティング（可能なら **ECMP** で2経路）
  5. **サービス**: サーバVLANに **LB + 複数バックエンド**（HTTP）を配置し、クライアントからLB経由でアクセス
  6. **セキュリティ**: クライアントからサーバへの **不要なICMPをACLで遮断**、HTTP(80)は許可
- **✅ 達成条件（すべて満たす）**:
  - [ ] クライアントがDHCPでIP取得できる
  - [ ] VLAN間・セグメント間がOSPF/SVIで疎通する
  - [ ] GWを1台落としてもVRRPで通信が継続する
  - [ ] LB経由のHTTPが複数バックエンドに分散され、1台落とすと自動迂回する
  - [ ] ACLでICMPが遮断され、HTTPは通る
